## 二、核心状态管理机制 (Store)

### 2.1 Store 核心API

[Store.js](file:///Users/tog/Desktop/code/solo/xyj-124/js/store/Store.js) 提供以下关键方法：

| 方法 | 用途 |
|------|------|
| `get(key)` | 读取指定状态 |
| `set(key, value)` | 直接设置状态，触发通知 |
| `update(key, updater)` | 函数式更新状态，触发通知 |
| `batch(fn)` | 批量更新，合并多次通知 |
| `subscribe(key, callback)` | 订阅指定 key 的变更 |
| `onAny(callback)` | 订阅所有状态变更 |
| `persist()` | 将状态保存到 localStorage |
| `restore()` | 从 localStorage 恢复状态 |

### 2.2 状态变更通知机制

```javascript
// Store.set() 简化流程
set(key, value) {
    const oldValue = this._state[key];
    this._state[key] = value;
    
    if (this._batchDepth > 0) {
        // 批量模式：暂存通知
        this._pendingNotifications.push({ key, value, oldValue });
    } else {
        // 非批量模式：立即通知
        this._notify(key, value, oldValue);
    }
}

// 通知分发
_notify(key, value, oldValue) {
    // 1. 通知指定key的订阅者
    (this._listeners[key] || []).forEach(cb => cb(value, oldValue));
    // 2. 通知全局订阅者（所有变更都会触发）
    this._globalListeners.forEach(cb => cb(key, value, oldValue));
}
```

### 2.3 全局订阅配置

在 [app.js#L137-L142](file:///Users/tog/Desktop/code/solo/xyj-124/js/app.js#L137-L142) 中配置了全局订阅：

```javascript
_setupStoreSubscriptions() {
    this.store.onAny(() => {
        this.store.persist();       // 1. 持久化到 localStorage
        this._renderAll();          // 2. 全量重新渲染所有模块
    });
}
```

> **关键特性**：任何状态变更都会触发 `_renderAll()`，这是数据驱动视图的核心机制。

### 4.5 批处理优化场景

当删除成员时，需要同时更新多个状态，使用 `batch()` 合并通知：

[MemberService.js#L35-L45](file:///Users/tog/Desktop/code/solo/xyj-124/js/services/MemberService.js#L35-L45)
```javascript
delete(id) {
    this.store.batch(() => {
        this.store.update('members', members => members.filter(m => m.id !== id));
        this.store.update('records', records => records.filter(r => r.memberId !== id));
        this.store.update('schedules', schedules => schedules.filter(s => s.memberId !== id));
        this.store.update('bills', bills => bills.filter(b => b.payerId !== id));
        this.store.update('settlements', settlements =>
            (settlements || []).filter(s => s.fromId !== id && s.toId !== id)
        );
    });
    // batch 结束后才会统一触发一次通知，避免多次渲染
}
```

**优化效果**：5次状态变更只触发1次通知，减少4次不必要的全量渲染。

## 五、数据变更到界面更新的完整机制

### 5.1 响应式更新原理

```
用户操作
    ↓
模块方法处理业务逻辑
    ↓
调用 Service 方法
    ↓
Service 调用 Store.set() / Store.update()
    ↓
Store 更新 _state 数据
    ↓
Store 调用 _notify() 分发变更通知
    ↓
全局订阅者 app.onAny() 被触发
    ├─→ store.persist() → 保存到 localStorage
    └─→ app._renderAll() → 全模块渲染
        ├─→ dashboardModule.render()
        ├─→ membersModule.render()
        ├─→ recordsModule.render()
        ├─→ scheduleModule.render()
        ├─→ inventoryModule.render()
        ├─→ billsModule.render()
        ├─→ votesModule.render()
        ├─→ messagesModule.render()
        ├─→ remindersModule.render()
        └─→ taskTypesModule.render()
            ↓
            每个 render() 方法：
            1. 从 Store 读取最新数据
            2. 根据数据生成 HTML 字符串
            3. 用 innerHTML 替换对应容器内容
    ↓
✅ DOM 更新 → 用户看到新界面
```

### 5.2 渲染性能分析

| 操作 | 触发渲染次数 | 影响范围 |
|------|-------------|----------|
| 添加成员 | 1次 | 所有模块 |
| 添加账单 | 1次 | 所有模块 |
| 删除成员（批处理） | 1次 | 所有模块 |
| 库存购买联动账单 | 2-3次 | 所有模块 |

> **注意**：本应用采用「全量重新渲染」策略，优点是逻辑简单、状态一致性高；缺点是性能开销较大。对于当前规模的应用（数据量小、DOM节点少）是可接受的。

### 5.3 状态一致性保证

1. **单一数据源**：所有数据存储在 Store._state 中，模块不保存本地状态
2. **单向数据流**：数据只能通过 Store.set/update 变更，不能直接修改
3. **同步更新**：状态变更和通知分发都是同步执行，避免竞态条件
4. **持久化时机**：每次状态变更后立即持久化，确保数据不丢失

## 七、架构总结与优化建议

### 7.1 架构优点

1. **清晰的分层**：模块、服务、状态管理职责明确
2. **单向数据流**：易于追踪数据流向，调试方便
3. **模块化设计**：各功能独立，便于维护和扩展
4. **服务层抽象**：业务逻辑可复用，测试友好
5. **状态一致性**：单一数据源避免数据不同步问题

### 7.2 潜在优化点

| 优化方向 | 当前实现 | 优化建议 |
|---------|---------|---------|
| **渲染性能** | 全量 innerHTML 重绘 | 可引入虚拟DOM或细粒度订阅，只渲染变更部分 |
| **状态变更追踪** | 无中间件机制 | 可添加日志中间件，便于调试状态变更 |
| **类型安全** | 纯JavaScript无类型检查 | 可迁移TypeScript或添加JSDoc类型注解 |
| **错误处理** | 缺少全局错误捕获 | 可添加全局异常处理器，避免操作失败白屏 |
| **表单验证** | 模块内分散验证 | 可抽象统一的表单验证工具类 |
